Document 2 — Detailed Correction Plan
I've organized this by priority. Each item follows the same pattern: what's wrong → what it should be → how to implement → result after implementation. This is the order Claude Code should tackle them.
Correction 1: Filing-Type Matching (THE blocker)
What's wrong. The Quiet Change pipeline pulls prev_text and curr_text without checking that both come from the same type of filing. Japanese disclosure uses several filing types with legally mandated but structurally different contents — 有価証券報告書 (annual securities report), 四半期報告書 (quarterly report), 決算短信 (earnings summary), and 半期報告書 (semi-annual report). Comparing any of these to any other will produce large text_change and semantic_change scores that reflect the filing type difference, not any change in the business. The 3923 ラクス pick in the current report is exactly this failure.
What it should be. The system must compare like-to-like: annual-to-annual from the prior year, Q2-to-Q2 from the prior year, etc. The senior's spec explicitly names 決算短信 as the intended source; the cleanest fix is to restrict the entire pipeline to 決算短信 at the same fiscal quarter in the prior year.
How to implement. In the ingest node, add a filter: for each company, identify the filing type and fiscal period of curr_text, then pull prev_text from the same filing type at the same fiscal period one year earlier. If the prior-year counterpart is missing or a different type, skip that company rather than flag it. Add a metadata field to the pipeline (filing_type, fiscal_period) so downstream nodes can verify the comparison is valid.
Result after. The composite_change score will no longer spike for structural reasons. Many currently-flagged picks will drop out. The remaining flags will be narrower but much more trustworthy. Expect the number of Quiet Change alerts to decrease significantly — this is correct behavior, not a regression.
Correction 2: Section-Level Comparison
What's wrong. The current code treats the filing as one blob of text. Cohen, Malloy & Nguyen's Lazy Prices paper — the exact paper the spec says must be read before starting — explicitly finds that changes in specific sections (executive discussion, risk factors, litigation) carry far more signal than whole-document change. Treating the filing as one blob dilutes the signal and makes the score more sensitive to boilerplate movement.
What it should be. Parse the filing into its named sections (for 決算短信: 経営成績, 財政状態, 今後の見通し, セグメント情報, etc.) and compute text_change and semantic_change per section. Aggregate with higher weights on the sections the literature says matter most (forward-looking statements, management discussion, risk discussion).
How to implement. Build a section parser that uses the standardized headings in 決算短信 as anchors. For each matched section in prev_text and curr_text, run the existing text_change and semantic_change measures. Keep a per-section score dictionary. The composite score becomes a weighted sum: composite = Σ w_section × change_section, with weights calibrated from backtest (start with equal weights as a placeholder until backtest data exists).
Result after. The system will distinguish "management changed their forward-looking statements" (strong signal) from "segment tables were reformatted" (no signal). Composite scores will be more interpretable — you'll be able to say which section drove the flag, not just that the filing changed.
Correction 3: Market Model CAR
What's wrong. The current CAR computation is stock_return - market_return — the "market-adjusted model." This implicitly assumes every stock has β = 1. For mid-cap Japanese stocks, βs range widely (often 0.6 to 1.5). When the Nikkei moves +2%, a β = 1.5 stock's "expected" move is +3%, not +2%. The current code would label +3% as +1% abnormal when it's actually zero abnormal. This is a known, documented bias — the event-study literature moved past this 40 years ago.
What it should be. The standard is the market model: R_i,t = α_i + β_i · R_m,t + ε, with α and β estimated over an estimation window before the event (convention: 120 to 250 trading days ending 30 days before the filing). Abnormal return for each event day is then R_i,t − (α_i + β_i · R_m,t).
How to implement. In car.py: for each company, before computing event-window abnormal returns, pull 150 trading days of returns ending 30 days before the filing date. Regress the stock's daily returns on the Nikkei's daily returns over that window to estimate α and β. Use those estimates in the event window. If the estimation window has fewer than 100 valid trading days, skip the company (or fall back to market-adjusted with a warning logged). Also switch the benchmark from Nikkei 225 to TOPIX, because the spec specifies TOPIX and because TOPIX is broader and more appropriate for mid-cap stocks.
Result after. Classifications in the 2×2 matrix will shift. Some current "Already Priced In" picks will become "Quiet Change" once the β correction reveals the price move was actually expected given market conditions. Some current "Quiet Change" picks will become "Already Priced In." Either shift is a correction of a real bias — this is the event-study equivalent of using proper units.
Correction 4: Larger, Industry-Gated Template Library
What's wrong. The Similar Company code uses three templates (8308, 6630, 3612) taken from the spec's examples as if they were the full library. No industry context is applied, so a SaaS growth company (ラクス) is compared against a beauty device crisis template (ヤーマン) and produces a "match" that is structurally meaningless. The spec warned about exactly this ("similar pattern, totally different context" mismatching).
What it should be. Two changes:
Expand the library. Compile a set of 20–40 historical Japanese mid-cap transformations from the past 15–20 years. Include multiple examples of each archetype (gradual improvement, crisis-forced recovery, growth-driven pivot, and likely 2–3 more archetypes the data will reveal). The spec's three named companies are a starting point, not the full set.
Industry-gate the matching. Before running DTW, filter eligible templates to those in a similar product/business space as the candidate. The spec suggests Hoberg-Phillips for this; a lightweight implementation uses cosine similarity of 事業内容 (business description) text from the 有価証券報告書.
How to implement. Build a template ingestion script that takes a list of historical tickers and pulls their segment revenue series during their transformation periods. Store each template with metadata: archetype name, industry sector, date range. Before the DTW loop in similar_company.py, filter templates to those whose business-description cosine similarity to the candidate exceeds some threshold (say, 0.35). Only compute DTW against the filtered template set.
Result after. The noise matches will disappear (ラクス will no longer "match" ヤーマン at 100% phase). Distances will spread out more — the best matches will actually stand out from the crowd. The few matches that survive industry gating will be far more actionable.
Correction 5: Diagnose the Phase-Progress Cluster
What's wrong. In the current report, 13 of 17 matches have phase progress between 80% and 100%. This is either a bug (partial DTW always aligning to the end of the template) or a look-ahead problem (templates all end at "now," so candidates always match the end segment).
What it should be. Phase progress should spread across the template duration — some candidates 20% in, some 60% in, some near completion. A healthy distribution is the diagnostic signal that partial DTW is working.
How to implement. Write a small diagnostic script: for each of the three current templates, generate 50 synthetic time series that are known to be at 25%, 50%, and 75% through the template (just truncate the template at those points with slight noise), then run the phase-progress calculation on them. If the calculated phases don't cluster around the true phases, the algorithm has a bug. Separately, check whether the template endpoints all correspond to recent dates — if yes, consider re-cutting templates to end at the historical moment the transformation completed, not at "today."
Result after. You will know whether phase progress is a usable decision input or not. If it's broken, fix it; if it's working, the information about "how early is this pattern" becomes valuable for sizing positions.
Correction 6: Replace Intersection With a Ranked Composite
What's wrong. The current aggregation is a strict AND: a company must be flagged by both detectors with hard thresholds to become a "high-conviction pick." This misses companies that are very strong in one signal and marginal in the other, and it treats the thresholds as knife-edges when the underlying quantities are continuous.
What it should be. A standardized composite score that combines both detectors continuously. For each candidate, compute a z-score of composite_change within the current batch, a z-score of (1 − min_DTW_distance) within the batch, and a z-score of (1 − |CAR|) within the batch. Sum these (or weight them by backtest-calibrated weights once those exist). Rank candidates by the composite; the top N become the actionable list.
How to implement. In the aggregate node: collect all detector outputs into a single table. Compute the three z-scores across the batch. Sum into final_score. Sort descending. Return the top N and also return the full ranked list so reviewers can see where the cutoff sits.
Result after. No more knife-edge misses. A company at composite 0.19 and DTW 0.11 will now be surfaced. Scoring is continuous and interpretable. The report will show both the top picks and how much better they are than the runners-up.
Correction 7: The Backtest
What's wrong. The spec allocates 1–2 weeks to backtesting. No backtest has been run. Without it, the direction of the signal is unknown. The Lazy Prices literature found that changers in the US underperform on average. If your version of the signal has the same sign in Japan, every "Quiet Change" flag is a sell signal, not a buy signal. You cannot tell until you check.
What it should be. A historical simulation: pick a cutoff date (say, end of 2022 or end of 2023, early enough that you have forward returns to measure). Run the full pipeline on filings up to that date. Record the flagged high-conviction picks. Measure their forward returns over 1, 3, 6, and 12 months against TOPIX. Compare against a control group of unflagged companies from the same universe and period.
How to implement. Add a backtest_mode flag to the pipeline. When set, it accepts a cutoff_date and only uses data available before that date. Run on historical filings from a 2–3 year window. Save flagged picks with their flag date. Separately, pull forward returns for each flagged company. Compute mean forward return, compute t-statistic against zero, compute mean forward return vs. the TOPIX benchmark. Produce a backtest report showing: total flags, mean forward return, vs-benchmark return, hit rate (fraction with positive vs-benchmark returns), and distribution.
Result after. You will know whether your system actually identifies outperformers. You will know whether the current thresholds are too strict, too loose, or approximately right. You will have calibrated weights for the composite score. You can then have a defensible conversation with your senior about what the system actually does and whether it should be deployed.
Deferred: The Qualitative Layer
Leadership changes, tone shifts, catalyst events, and other qualitative features are genuinely valuable additions — the academic literature on CEO turnover, managerial tone, and catalyst detection is strong. But they should be added after Corrections 1–7 are complete. Adding them now would layer sophistication on a broken foundation and obscure which bug is causing what problem. I'm listing them here so they don't get lost, but Claude Code should not touch them in the current round.
